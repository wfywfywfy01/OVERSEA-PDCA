# M9 开发复盘 · 核心经验 12 条

## 技术层面

**单文件原型的黄金三角：Alpine.js + Tailwind + localStorage**
不要上构建工具。三者组合足以支撑中等复杂度的管理系统原型，CDN 引入即用，零配置，演示时打开 HTML 就跑。

**数据版本管理从第一天就加**
改数据结构就升 `DATA_VERSION`，旧缓存自动失效重新注入 Mock。不加这个，结构变更后浏览器用旧数据渲染新逻辑，产生"没改任何东西但界面不对"的玄学 bug，极难定位。

**Alpine.js 里 table + 嵌套 x-for 会出问题**
改成扁平数组 + 单层 `x-for`。把树状结构（组 → 成员）在 JS 侧展开成带 `type: 'group' | 'member'` 标记的行数组，模板只做一层循环，彻底规避 Alpine 对嵌套表格渲染的处理缺陷。

## 产品层面

**时间维度的核心设计原则：层级越高看的周期越长**
销售看日/周（短期行动反馈），组长看周/月（节奏管控），总监看月/季（趋势判断）。切换器按角色权限过滤可选粒度，不要让销售看季度数据——对他们没有行动意义。

**累计指标和本期指标要分两行展示**
成交总数（累计）和本期触达次数（本期）语义完全不同，混在同一个数字里会让用户不知道看的是什么。分两行：第一行本期，第二行累计，标注清楚。

**Mock 数据必须在四个时间层次都有数据**
今日 / 本周 / 本月 / 本季各层次都要埋数据点，否则切换维度时全是空状态，无法验证交互逻辑，演示时也没有感觉。

## 工程层面

**角色判断要封装成语义化方法**
`isDirector()` / `isTeamLead()` / `hasManageView()` 集中定义，不要在模板和业务函数里散落写 `role === 'director'`。角色枚举一旦变更，散落的判断会漏改，封装后只改一处。

**Hermes 这类 AI 助手要做成"可交互行动卡片"**
每条洞察消息必须带 `action`（跳转 tab + 可选重置筛选器），纯展示文字没有用。用户点击后有明确去处，才能形成"看到异常 → 点击 → 直达问题"的闭环，否则 AI 提醒只是装饰。
